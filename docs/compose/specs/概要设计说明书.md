## 2. 系统总体设计

### 2.1 设计原则

| 原则 | 说明 |
|------|------|
| 前后端分离 | 前端和后端独立开发、独立部署，通过API接口通信 |
| 模块化设计 | 系统按功能划分为独立模块，模块间通过接口交互 |
| 高内聚低耦合 | 模块内部功能紧密相关，模块间依赖最小化 |
| 可扩展性 | 预留扩展接口，便于后续功能扩展 |
| 安全性 | 数据加密传输、密码哈希存储、接口鉴权 |

## 3. 模块详细设计

### 3.1 认证模块 (Auth)

#### 3.1.1 功能说明

处理用户的注册、登录、身份验证等功能。

#### 3.1.2 模块接口

| 接口 | 方法 | 描述 | 请求参数 | 响应数据 |
|------|------|------|---------|---------|
| /api/auth/register | POST | 用户注册 | username, password, name, school, subject, grade, role | user_info, token |
| /api/auth/login | POST | 用户登录 | username, password | user_info, token |
| /api/auth/profile | GET | 获取用户信息 | - | user_info |
| /api/auth/update | PUT | 更新用户信息 | user_info | user_info |

#### 3.1.3 认证流程

```
登录流程：
┌──────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 用户  │───▶│ 前端页面  │───▶│ 后端接口  │───▶│ 数据库    │
│      │◀───│ 登录表单  │◀───│ 验证逻辑  │◀───│ 用户表    │
└──────┘    └──────────┘    └──────────┘    └──────────┘

1. 用户输入用户名和密码
2. 前端发送POST请求到 /api/auth/login
3. 后端验证用户名和密码（密码哈希比对）
4. 验证成功 → 生成JWT Token → 返回用户信息和Token
5. 前端保存Token到localStorage → 跳转到对应页面

Token验证流程：
┌──────────┐    ┌──────────┐    ┌──────────┐
│ 前端请求  │───▶│ Token校验 │───▶│ 后端接口  │
│ (带Token) │    │ 中间件    │    │ 业务处理  │
└──────────┘    └──────────┘    └──────────┘

1. 前端每次请求携带Token（Authorization: Bearer <token>）
2. 后端中间件验证Token有效性
3. 验证通过 → 提取用户信息 → 执行业务逻辑
4. 验证失败 → 返回401未授权
```

#### 3.1.4 安全设计

| 安全措施 | 实现方式 |
|---------|---------|
| 密码加密 | bcrypt哈希算法 |
| Token认证 | JWT（JSON Web Token） |
| Token有效期 | 24小时 |
| 接口鉴权 | FastAPI依赖注入 |

### 3.2 工具管理模块 (ToolMgmt)

#### 3.2.1 功能说明

管理AI智能体工具的增删改查，包括工具创建、编辑、删除、分享等功能。

#### 3.2.2 模块接口

| 接口 | 方法 | 描述 | 请求参数 | 响应数据 |
|------|------|------|---------|---------|
| /api/tools | GET | 获取工具列表 | category, subject, page, limit | tools[], total |
| /api/tools/{id} | GET | 获取工具详情 | id | tool_info |
| /api/tools | POST | 创建工具 | tool_info | tool_info |
| /api/tools/{id} | PUT | 更新工具 | id, tool_info | tool_info |
| /api/tools/{id} | DELETE | 删除工具 | id | success |
| /api/tools/{id}/share | GET | 生成分享链接 | id | share_url, qr_code |
| /api/tools/presets | GET | 获取预设工具 | category, subject | tools[] |
| /api/tools/recommend | GET | 推荐工具 | category, need_type | tools[] |

#### 3.2.3 工具创建流程

```
AI对话创建工具流程：

┌──────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│ 教师  │──▶│ 前端页面  │──▶│ 后端接口  │──▶│ AI服务   │──▶│ 数据库    │
│      │◀──│ AI对话框  │◀──│ 创建逻辑  │◀──│ DeepSeek │◀──│ 工具表    │
└──────┘   └──────────┘   └──────────┘   └──────────┘   └──────────┘

1. 教师描述工具需求（如"我想创建一个古诗词赏析工具"）
2. 后端调用DeepSeek API，使用创建工具的提示词模板
3. DeepSeek返回结构化的工具配置（名称、描述、提示词、参数）
4. 后端解析返回内容，生成工具配置
5. 教师确认 → 保存到数据库
6. 生成分享链接/二维码
```

#### 3.2.4 分享机制

```
分享链接生成：

1. 为工具生成唯一分享码（share_code）
2. 拼接分享URL：https://域名/tool/{share_code}
3. 使用qrcode库生成二维码图片
4. 返回分享URL和二维码图片数据

使用记录：
- 学生通过分享链接访问 → 无需登录
- 后端记录使用日志（tool_id, input, output, session_id）
- 匿名用户的user_id为NULL
```

### 3.3 AI对话模块 (AI)

#### 3.3.1 功能说明

处理与DeepSeek API的交互，实现文本生成。

#### 3.3.2 模块接口

| 接口 | 方法 | 描述 | 请求参数 | 响应数据 |
|------|------|------|---------|---------|
| /api/chat | POST | 发送消息 | tool_id, message, session_id | reply, session_id |
| /api/chat/stream | POST | 流式返回 | tool_id, message, session_id | SSE stream |

#### 3.3.3 AI对话流程

```
AI对话处理流程：

┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 前端发送  │───▶│ 后端接收  │───▶│ 读取配置  │───▶│ 拼接提示  │
│ 用户消息  │    │ 请求     │    │ 提示词    │    │ 词       │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
                                                       │
                                                       ▼
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 前端展示  │◀───│ 后端返回  │◀───│ DeepSeek │◀───│ 调用API  │
│ AI回复   │    │ 回复内容  │    │ API处理  │    │ 发送请求  │
└──────────┘    └──────────┘    └──────────┘    └──────────┘

详细步骤：
1. 前端发送用户消息和工具ID
2. 后端根据tool_id从数据库读取工具的提示词模板
3. 拼接系统提示词 + 对话历史 + 用户消息
4. 调用DeepSeek API（model: dsv4）
5. 获取AI回复内容
6. 记录使用日志（input, output）
7. 返回回复内容给前端
8. 前端渲染展示（支持Markdown格式）
```

#### 3.3.4 提示词管理

```
提示词模板结构：

{
  "system_prompt": "你是一个古诗词趣味赏析助手...",
  "input_label": "请输入你想了解的古诗词",
  "output_format": "markdown",
  "parameters": {
    "temperature": 0.7,
    "max_tokens": 2000
  }
}

提示词来源：
1. 预设工具：数据库中预存的提示词模板
2. 教师创建：教师通过AI对话生成的提示词
3. 通用问答：默认的AI学习助手提示词
```

### 3.4 推荐引擎模块 (Recommend)

#### 3.4.1 功能说明

基于条件匹配的简单推荐系统，根据教师选择的学科和需求类型推荐合适的工具。

#### 3.4.2 推荐逻辑

```
推荐规则表结构：

┌──────────┬──────────┬──────────┬──────────┐
│ 学科     │ 需求类型  │ 推荐工具  │ 优先级    │
├──────────┼──────────┼──────────┼──────────┤
│ 文科     │ 课前备课  │ [1,2]   │ 0        │
│ 文科     │ 课堂教学  │ [1,3]   │ 0        │
│ 理科     │ 课前备课  │ [7,8]   │ 0        │
│ 理科     │ 课堂教学  │ [7,10]  │ 0        │
└──────────┴──────────┴──────────┴──────────┘

推荐算法（if-else条件匹配）：

if 文科:
    if 课前备课: 推荐 [古诗词赏析, 作文灵感助手]
    if 课堂教学: 推荐 [古诗词赏析, 阅读理解辅助]
    if 课后辅导: 推荐 [名词解释器, 阅读理解辅助]
    if 兴趣激发: 推荐 [古诗词赏析, 诗词飞花令]
if 理科:
    if 课前备课: 推荐 [公式推导, 实验模拟]
    if 课堂教学: 推荐 [公式推导, 逻辑训练]
    if 课后辅导: 推荐 [错题分析, 概念辨析]
    if 兴趣激发: 推荐 [逻辑训练, 数据分析]
```

#### 3.4.3 推荐接口

| 接口 | 方法 | 描述 | 请求参数 | 响应数据 |
|------|------|------|---------|---------|
| /api/tools/recommend | GET | 获取推荐工具 | category, need_type | tools[] |

#### 3.4.4 推荐流程

```
推荐流程：

1. 教师选择文科/理科 → 前端传递category参数
2. 教师选择需求类型 → 前端传递need_type参数
3. 后端查询推荐规则表 → 获取匹配的tool_ids
4. 根据tool_ids查询工具详情
5. 按优先级排序 → 返回推荐工具列表
6. 前端展示推荐工具
```

### 6.1 认证与授权

| 安全措施 | 实现方式 |

| 密码存储 | bcrypt哈希加密 |

### 6.2 数据安全

| 安全措施 | 实现方式 |
|---------|---------|

| XSS防护 | 前端输入过滤、后端输出编码 |
| CSRF防护 | Token验证 |
| 数据传输 | HTTPS加密（部署时配置） |

### 6.3 API安全

| 安全措施 | 实现方式 |
|---------|---------|
| 请求限流 | FastAPI中间件 |
| 参数校验 | Pydantic数据模型 |
| 错误信息 | 统一错误响应，不暴露内部信息 |
