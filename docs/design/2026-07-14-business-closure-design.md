# 智教通业务闭环设计

## 目标

将智教通从可展示的前端和接口骨架升级为可在本机复现的教师—学生业务闭环：教师登录后管理自己的工具并分享，学生通过公开链接或工具广场使用工具，AI 对话和使用记录准确归属到登录用户，教师可以查看自己的数据。

## 本地运行边界

本机使用已安装的 MySQL 8.4。创建专用数据库 `zjiaotong` 与最小权限账号 `zjiaotong_app`，其随机密码只写入忽略的 `source/backend/.env`。不再使用硬编码的 `root/admin` 连接。

`source/backend/.env.example` 只保留变量名和示例值；真实 MySQL 密码、JWT 密钥和 DeepSeek Key 永不进入 Git、日志或前端构建产物。

## 身份与权限

- 使用 bcrypt 哈希密码，移除固定 salt 的 SHA-256。
- 前端 Axios 统一附带 `Authorization: Bearer <token>`。
- 后端新增当前用户依赖，令牌失效或缺失返回 401，角色不匹配返回 403。
- 教师只能创建、修改、删除自己创建的非预设工具；学生仅可访问公开工具。
- 个人资料、学习记录和教师看板不再接受可伪造的 `user_id` 查询参数，而是从令牌推导用户身份。

## 业务数据流

```text
教师登录 → 需求引导 → 推荐工具 → 创建/管理工具 → 生成公开分享链接
                                                  ↓
学生注册/登录或打开公开链接 → 使用工具/AI问答 → UsageLog(user_id, tool_id)
                                                  ↓
学生学习记录 ← 真实个人日志       教师看板 ← 自己的工具使用数据
```

兴趣引导将三层选择编码为明确的查询参数；工具广场读取这些参数并调用推荐接口或进行与学科、困难、学习方式一致的本地筛选。公开分享统一使用 `/share/:shareCode` 和 `GET /api/tools/share/:shareCode`，不再生成不存在的 `/student/tool/:shareCode` 地址。

## AI 边界

DeepSeek 服务有两种显式模式：

- `AI_PROVIDER=mock`：无 Key 的本地开发和自动化测试，响应标记为模拟结果。
- `AI_PROVIDER=deepseek`：必须同时存在真实 `DEEPSEEK_API_KEY` 与允许的模型名；缺失配置返回可理解的 503，不伪装为真实 AI。

真实 DeepSeek 请求只能在用户把 Key 写入本地 `.env` 后运行。健康检查会报告数据库与 AI 配置状态，但绝不返回密钥。

## API 与前端调整

- 认证：登录响应统一返回 `access_token`、用户资料；前端 store 使用同一契约。
- 工具：增加“我的工具”、公开分享查询、所有权检查与创建表单。
- 聊天：`user_id` 由 JWT 写入 `UsageLog`，无工具的通用问答使用可追踪的系统工具而非硬编码 ID。
- 记录：前端从登录状态读取身份；后端只返回当前用户记录。
- 看板：教师只看自己创建的工具及其使用数据。
- 推荐：学生兴趣参数和教师需求参数都映射到公开工具筛选。

## 验证策略

先以 pytest + SQLite 测试数据库覆盖认证、权限、分享、日志归属、推荐与 AI 模式；再用真实 MySQL 执行初始化与 API smoke test；最后启动 FastAPI 与 Vite，执行教师—学生完整 API 流程。前端单元测试、后端测试、Vite 构建和真实 MySQL smoke test 全部通过后才提交。

## 不在本次范围

- 不把师大智能体平台的“创建智能体”能力复制进本应用。
- 不把 DeepSeek Key 放入前端。
- 不重写已经完成的东方未来主义视觉系统。
- 不进行公网部署；本次交付先保证本机可复现闭环，并保留后续部署所需配置接口。
