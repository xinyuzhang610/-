# 智教通项目模块化架构规范

> 日期：2026-07-13
> 目的：制定代码架构规范，避免"屎山代码"，预留扩展空间

## [S1] 项目核心理念（技术规范的前提）

**项目目标**：为零基础小白教会使用AI工具并挖掘兴趣。

**技术规范的定位**：服务于这个目标，让项目能够持续迭代、易于维护，最终交付一个可用的产品。

## [S2] 技术架构原则

**核心目标**：代码易读、易改、易扩展，不会变成"屎山"。

**四大技术原则**：
1. **模块独立**：每个功能独立成模块，改一个不影响其他
2. **接口清晰**：模块之间通过固定格式通信，不乱调用
3. **配置外部**：常量、阈值放配置文件，改参数不改代码
4. **文档同步**：每个模块有简要说明，新人能快速上手

## [S3] 前端架构规范

**目录结构**：
```
src/
├── api/          # API调用层（每个业务模块一个文件）
├── components/   # 公共组件（可复用的UI组件）
├── composables/  # 组合式函数（可复用的逻辑）
├── layouts/      # 布局组件（教师端/学生端布局）
├── stores/       # 状态管理（Pinia）
├── views/        # 页面组件
│   ├── auth/     # 登录注册
│   ├── teacher/  # 教师端页面
│   └── student/  # 学生端页面
└── utils/        # 工具函数
```

**代码规范**：
1. **组件拆分**：单个 `.vue` 文件不超过 300 行，超过就拆分成子组件
2. **API集中管理**：所有 API 调用放在 `api/` 目录，页面不直接写 axios
3. **状态管理**：用户信息、登录状态放 `stores/user.js`，页面间共享
4. **样式隔离**：每个组件用 `<style scoped>`，避免样式冲突

**扩展预留**：
- 新增页面：在 `views/` 下新建文件夹
- 新增组件：在 `components/` 下新建文件
- 新增逻辑：在 `composables/` 下新建函数

## [S4] 后端架构规范

**目录结构**：
```
source/backend/
├── api/          # API路由层（每个业务模块一个文件）
├── models/       # 数据模型（SQLAlchemy ORM）
├── schemas/      # 数据验证（Pydantic）
├── services/     # 业务逻辑层（核心逻辑）
├── utils/        # 工具函数
├── config.py     # 配置文件
└── main.py       # 入口文件
```

**代码规范**：
1. **三层架构**：路由层(api) → 业务逻辑层(services) → 数据层(models)
2. **路由只做转发**：`api/` 文件只处理请求/响应，不写业务逻辑
3. **业务逻辑在 services**：所有核心逻辑放在 `services/` 对应文件
4. **数据验证用 schemas**：请求/响应数据格式统一用 Pydantic 校验

**扩展预留**：
- 新增 API：在 `api/` 下新建文件，在 `main.py` 注册路由
- 新增业务逻辑：在 `services/` 下新建文件
- 新增数据模型：在 `models/` 下新建类

## [S5] 数据库架构规范

**设计原则**：
1. **字段留余**：核心表预留 `ext_info` JSON 字段，存扩展属性
2. **软删除**：用 `is_deleted` 字段代替真删除，方便恢复
3. **时间戳**：所有表都有 `created_at` 和 `updated_at`
4. **索引优化**：高频查询字段加索引，但不过度索引

**扩展预留**：
- 新增属性：存入 `ext_info` JSON 字段，不用改表结构
- 新增关联：通过外键关联新表，不用改现有表
- 数据迁移：SQL 文件放在 `sql/` 目录，版本化管理

## [S6] 配置管理规范

**配置文件**：
- `config.py`：Python 配置类，集中管理所有配置
- `.env`：环境变量（API Key、数据库密码等敏感信息）
- `vite.config.js`：前端配置（代理、构建等）

**配置原则**：
1. **敏感信息放 .env**：API Key、密码等不进代码库
2. **业务常量放 config.py**：如分页大小、超时时间等
3. **环境切换用环境变量**：开发/测试/生产用不同配置

## [S7] 文档规范

**文档分类**：
1. **代码注释**：关键函数写 1-2 行注释说明用途
2. **模块说明**：每个目录下的 `README.md`（可选，复杂模块才需要）
3. **API 文档**：Swagger UI 自动生成（已配置）
4. **竞赛文档**：已完成的 6 份文档

**文档原则**：
1. **注释写"为什么"**：不写"这段代码做了什么"，写"为什么这样做"
2. **文档跟代码走**：文档放在代码旁边，不要单独放
3. **及时更新**：改代码时同步更新相关文档
